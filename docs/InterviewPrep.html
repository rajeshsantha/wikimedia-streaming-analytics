System Design & Architecture
End-to-end pipeline design, failure modes, scaling
6 questions
Q Walk me through the end-to-end data flow in this pipeline.CORE ▼
ANSWER
SSE Feed→ OkHttp Client→ Kafka Producer→ wikimedia-events→ Spark Streaming→ 3 Queries→ Delta Lake

Stage 1 — Ingestion: A long-lived HTTP connection to the Wikimedia SSE endpoint streams newline-delimited text/event-stream events. OkHttp's SSE client parses these and forwards raw JSON payloads to a Kafka producer.

Stage 2 — Buffering: The producer writes to the wikimedia-events Kafka topic with acks=all and idempotence enabled. Kafka decouples the live SSE stream from downstream consumers and provides a durable, replayable log.

Stage 3 — Processing: Spark Structured Streaming reads the Kafka topic, deserializes JSON into a typed Dataset using a defined StructType schema, then fans out into three parallel streaming queries: sliding trending windows, stateful active-editor tracking via mapGroupsWithState, and a spike detector.

Stage 4 — Storage: Each query writes its output to a separate Delta Lake table. Delta's transaction log ensures atomic, exactly-once micro-batch commits. Checkpoints track Kafka offsets and Spark state so the pipeline can resume after any crash.

In interviews, draw this as a whiteboard diagram and walk left-to-right. Highlight where exactly-once guarantees are established at each boundary.
Q Why use Kafka between the SSE source and Spark?MEDIUM ▼
ANSWER

Directly connecting Spark to the SSE feed creates tight coupling — a Spark restart or slow batch would either drop events or stall the HTTP connection. Kafka solves this with five key properties:

PropertyWhy it matters here
Durable replayKafka retains events for a configurable retention window; Spark can re-read from its last committed offset after any failure without data loss.
Backpressure isolationIf Spark is slow (e.g., a large state checkpoint), it reads at its own pace while Kafka absorbs the incoming burst without backpressuring the SSE server.
Fan-outMultiple independent consumer groups (analytics, archiving, alerting) can each read the same topic independently — no re-ingestion needed.
Offset trackingCombined with Spark checkpoints, Kafka offsets give you a precise restart point for exactly-once processing semantics.
Operational decouplingThe SSE producer and Spark consumers can be deployed, restarted, or scaled independently.

Q How would you scale this from local to production?HARD ▼
ANSWER

Scaling touches every layer of the stack:

Kafka: Move from a single broker to a multi-broker cluster with replication.factor=3 and min.insync.replicas=2. Right-size partition count to match peak throughput ÷ per-partition throughput (~10–50 MB/s). Enable SASL/SSL for auth and encryption.

Spark: Deploy on Kubernetes (Spark Operator) or YARN. Enable dynamic executor allocation so the cluster scales up for burst traffic and scales down during quiet periods. Tune spark.sql.shuffle.partitions to 2–3× the cluster cores.

State Store: Replace the default in-memory store with RocksDB (spark.sql.streaming.stateStore.providerClass=RocksDBStateStoreProvider) to avoid driver OOM on large key cardinality.

Delta Lake: Use S3/GCS/ADLS as the backing store instead of local disk. Enable auto-optimize and auto-compaction to prevent small-file proliferation.

Observability: Expose Spark metrics to Prometheus via the JMX exporter and visualize with Grafana. Alert on Kafka consumer lag, batch duration P99, and state store size.

CI/CD: Build and publish Spark application JARs via GitHub Actions. Use ArgoCD or Flux for Kubernetes deployment.

F/U How would you handle a schema change in the Wikimedia feed without downtime?HARD ▼
FOLLOW-UP ANSWER

Schema evolution without downtime requires coordination across the producer, Spark, and Delta layers:

1. Schema Registry: In production, register all schemas in Confluent Schema Registry with BACKWARD or FULL compatibility mode. Producers validate before publishing; consumers can read older messages even after a schema bump.

2. Spark Schema Handling: Use from_json(col, schema, Map("mode" → "PERMISSIVE")) so new unknown fields are silently dropped and missing old fields produce nulls — the pipeline doesn't break.

3. Delta Evolution: Set .option("mergeSchema", "true") on the Delta write. Delta will add new columns to the table schema automatically. Downstream queries using SELECT * will immediately see new columns; typed queries remain unchanged.

4. Blue/Green Rollout: If the schema change is breaking (field rename, type change), deploy a second pipeline reading from a new Kafka topic, run both in parallel until confidence is established, then cut over downstream consumers.

The safest pattern is additive-only schema changes. Treat field removal or type changes as breaking changes requiring a versioned topic.
F/U If you had to add a second analytics pipeline consuming the same data, how would you design it? ▼
FOLLOW-UP ANSWER

The cleanest approach is a separate Kafka consumer group. Each consumer group maintains independent offsets, so the new pipeline reads from the beginning of the retention window without affecting the existing one.

In Spark, you'd add a new streaming query on the same parsed DataFrame (or re-read from Kafka with a different groupId). Each query gets its own checkpoint path and output Delta table. Shared transformation logic can be extracted into a library to avoid drift.

If the second pipeline needs the exact same parsed/cleaned data, consider writing to an intermediate "canonical" Delta table first, and having all downstream queries read from it — a "medallion architecture" (Bronze → Silver → Gold pattern).

Q How does the SSE client handle reconnects and missed events?MEDIUM ▼
ANSWER

OkHttp's SSE implementation handles reconnects automatically. On a network error or non-200 response, the client waits for a configurable retry delay and re-establishes the connection.

To limit data loss, the SSE spec defines the Last-Event-ID mechanism: every SSE event can carry an id: field. The client stores the most recently received ID and includes it as the Last-Event-ID HTTP header on reconnect. If the Wikimedia server supports replay (it does for short windows), it will re-stream events from that ID forward.

However, for events missed during longer outages, Kafka retention is the real safety net — the producer wrote those events before the client crashed, so they're replayable from the consumer side.

The combination of SSE Last-Event-ID (short gaps) + Kafka retention (long gaps) gives layered durability for the ingestion layer.
Q Why use acks=all and enable.idempotence=true together?MEDIUM ▼
ANSWER

These two settings together give you the strongest durability and exactly-once write guarantee at the producer level:

acks=all: The broker only acknowledges a message after all in-sync replicas (ISR) have written it to their local logs. This means even if the leader crashes immediately after acknowledgment, no data is lost — at least one follower has the message and can become the new leader.

enable.idempotence=true: The producer assigns each message a Producer ID (PID) and a per-partition sequence number. If a send is retried (e.g., due to a timeout before receiving an ack), the broker detects the duplicate sequence number and silently discards it — so retries never produce duplicate records.

Without idempotence, a network timeout could cause the producer to retry a send that actually succeeded, resulting in duplicates in the log.

// Java producer config equivalent
props.put("acks", "all");
props.put("enable.idempotence", "true");
props.put("max.in.flight.requests.per.connection", "5"); // safe with idempotence
props.put("retries", Integer.MAX_VALUE);
Enabling idempotence automatically sets acks=all and retries=MAX if not explicitly set. Explicit configuration is still recommended for clarity.
Q What's the impact of linger.ms and compression on latency and throughput?MEDIUM ▼
ANSWER

linger.ms controls how long the producer waits before sending a batch. The trade-off is:

At linger.ms=0 (default), every message is sent immediately — minimum latency, but small batches mean more network round trips and worse compression ratios.

At linger.ms=5–20, the producer accumulates messages into larger batches before sending. This dramatically improves throughput (fewer requests), reduces broker CPU, and improves compression. The cost is an added 5–20ms of latency per message — acceptable for analytics pipelines but not for low-latency financial systems.

Compression (Snappy/LZ4/Zstd): Applied per-batch at the producer, decompressed at the consumer. Benefits: 3–5× reduction in network bandwidth and broker disk usage. Costs: marginal CPU overhead on producer and consumer. Snappy favors speed; Zstd favors compression ratio. For Wikimedia's JSON payloads (highly compressible text), Snappy typically gives 60–70% size reduction.

ConfigLatencyThroughputCPU
linger=0, no compressionLowestLowestLowest
linger=10ms, Snappy+10msHighLow
linger=10ms, Zstd+10msHighestMedium
F/U What happens if the Kafka producer's buffer fills up — how do you handle back pressure?HARD ▼
FOLLOW-UP ANSWER

The producer's in-memory record accumulator has a fixed size (buffer.memory, default 32 MB). When it fills up — meaning the broker is slow to accept batches — subsequent producer.send() calls block for up to max.block.ms (default 60s). After that timeout, a TimeoutException is thrown.

For the Wikimedia ingestion use case, options are:

1. Increase buffer + linger: Raise buffer.memory to accommodate burst traffic. Pair with higher linger.ms to batch more aggressively.

2. Async send with callback: Use non-blocking send(record, callback) and track in-flight count. If the callback queue exceeds a threshold, pause the SSE listener thread and resume after draining.

3. Drop vs. block trade-off: For this pipeline, blocking (and potentially dropping SSE events) is acceptable because Kafka is the durability layer — events not yet produced are truly lost, but events already in Kafka are safe. For mission-critical systems, you'd buffer overflow to local disk.

A slow broker causing producer back pressure that blocks the SSE consumer thread will eventually trigger SSE reconnects. Monitor producer request latency (P99) as an early warning signal.
F/U How would you partition the Kafka topic to balance load while preserving ordering for per-article events? ▼
FOLLOW-UP ANSWER

Use the article title (or page ID) as the Kafka message key. Kafka's default partitioner hashes the key to a partition, so all edits for a given article land on the same partition in order. This enables Spark to compute per-article aggregations without cross-partition shuffles on the hot path.

The trade-off: a very popular article (e.g., a breaking news page) will generate a hot partition with disproportionate load. Mitigations include sticky partitioning for a burst window, or a two-level approach: partition by hash(articleId) % numPartitions but within Spark, re-aggregate with a downstream shuffle keyed on articleId.

Kafka Fundamentals
Partitions, ordering, exactly-once, KRaft
6 questions
Q Why choose a topic with multiple partitions, and how does Spark read them?EASY ▼
ANSWER

A single Kafka partition is a single ordered log — reads and writes are sequential, bound by one broker's I/O. Multiple partitions allow horizontal scaling of both write throughput (producer parallelism) and read throughput (consumer parallelism).

Spark's Kafka source creates one RDD partition (and thus one task) per Kafka partition per micro-batch. With 8 Kafka partitions and 8 executor cores, all partitions are read in parallel — 8× the throughput of a single partition.

Partition count should be sized as: max_throughput_MB_s ÷ per_partition_throughput_MB_s. A typical rule of thumb is 10–20 partitions per broker for moderate workloads.

You can only increase partitions, never decrease them. Pick a partition count that gives headroom for 12–18 months of growth at the time of topic creation.
Q How do you guarantee ordering, and when does ordering break down?MEDIUM ▼
ANSWER

Kafka guarantees strict ordering within a single partition. Producers append to the tail; consumers read sequentially. This means all messages with the same key (same article) arrive in write order if the producer uses consistent key-based partitioning.

Ordering breaks down in these scenarios:

1. Multiple partitions, no key: Round-robin assignment spreads messages across partitions. Consumer read order becomes interleaved and non-deterministic.

2. Multiple producers: If two producer instances write the same key simultaneously, network variability means the "later" message may land before the "earlier" one.

3. max.in.flight.requests > 1 without idempotence: Retried batches can overtake in-flight batches, causing reordering. Idempotent producers with sequence numbers prevent this.

4. Consumer parallelism: Even with ordered partitions, a multi-threaded consumer processing messages from the same partition in parallel loses ordering in the processing layer.

Q What would you change if you needed exactly-once from producer to sink?HARD ▼
ANSWER

End-to-end exactly-once requires guarantees at every stage in the chain:

Producer → Kafka: Enable idempotent producer (enable.idempotence=true) and use transactional APIs (transactional.id) when producing to multiple partitions atomically. This gives exactly-once publish semantics.

Kafka → Spark: Spark's Structured Streaming with Kafka uses offset-based checkpointing. Each micro-batch reads a precise offset range and commits only after the output is written. This gives exactly-once read semantics.

Spark → Sink: Use an idempotent or transactional sink. Delta Lake's transaction log with batch IDs provides exactly-once write semantics — Spark embeds a unique batch ID in every commit, and Delta rejects duplicate batch IDs on restart.

Consumer isolation: Set isolation.level=read_committed on any Kafka consumer that reads from transactionally produced topics, so it only sees committed (non-aborted) transactions.

// Transactional producer init
props.put("transactional.id", "wikimedia-producer-0");
producer.initTransactions();
producer.beginTransaction();
// ... send messages
producer.commitTransaction(); // atomic

F/U How would you size Kafka retention for this pipeline, and how does it interact with Spark's checkpoint?HARD ▼
FOLLOW-UP ANSWER

Kafka retention must be long enough to cover the worst-case Spark downtime — the period between the last committed Spark offset and when the pipeline is restarted. If Kafka deletes data before Spark can re-read it, the pipeline will fail with an offset-out-of-range error.

A practical formula: retention.ms = SLA for pipeline recovery × 2. If your ops team guarantees a 24-hour incident response, set retention to 48 hours minimum. Add headroom for schema debugging or replay.

The Spark checkpoint stores the last committed Kafka offset. On restart, Spark issues a seekToOffset call for that offset. If the offset has been deleted from Kafka (past retention), Spark will either fail or skip to the earliest available offset — causing a data gap. Setting failOnDataLoss=false allows skipping, but silently loses data.

Set failOnDataLoss=true in production to get an explicit failure instead of silent data loss when offsets expire.
F/U What is a consumer group rebalance, and how does it affect a streaming Spark job? ▼
FOLLOW-UP ANSWER

A consumer group rebalance is triggered when a consumer joins or leaves the group, or when a partition is added. During rebalance, all partition assignments are revoked, a new assignment is computed by the group coordinator, and partitions are redistributed — causing a brief processing pause.

For Spark Structured Streaming, the Kafka connector handles rebalances internally per micro-batch: at the start of each batch it checks the current assignment and adjusts offset ranges accordingly. If a rebalance causes a batch to be long-running, it may appear as a latency spike in the Spark UI.

Mitigation: Use assign mode instead of subscribe in Spark's Kafka source when you have a fixed, known partition count — this bypasses the group coordinator entirely and avoids rebalances.

Event time, watermarks, windows, mapGroupsWithState
7 questions
Q Explain event time vs processing time in this pipeline.EASY ▼
ANSWER

Event time is the timestamp embedded in the Wikimedia event payload — the moment the edit actually occurred on Wikipedia. Processing time is when Spark reads and processes that event.

The gap between them is the processing lag: events published minutes ago, queued in Kafka, and read in the current micro-batch will have event times significantly behind processing time. Using event time windows correctly places those events in the window they belong to, not the window they arrived in.

Q What does watermarking do, and what happens without it?MEDIUM ▼
ANSWER

A watermark is a threshold that tells Spark: "events more than X time behind the current max event time are considered too late and will be dropped." Spark advances the watermark as it sees newer event times, and uses it to trigger window finalizations and evict stale state.

Concretely: if Spark has seen events up to 10:00 PM and the watermark is 10 minutes, any event with a timestamp before 9:50 PM is dropped. Windows ending before 9:50 PM can be safely finalized and their state freed.

Without watermarking: Spark must retain state for every active window indefinitely, because a late event could always arrive and update an old window. Over hours of operation, state accumulates without bound and the job runs out of memory or disk on the state store.

// Example: 10-minute sliding window with 15-minute watermark
df.withWatermark("eventTime", "15 minutes")
  .groupBy(window(col("eventTime"), "10 minutes", "30 seconds"), col("title"))
  .count()
Set the watermark to be at least as large as your maximum expected event lateness. Too tight a watermark silently drops legitimate late events; too loose a watermark wastes memory retaining stale state.
Q Why use sliding windows for trending articles and tumbling windows for spike detection?MEDIUM ▼
ANSWER

Sliding windows (10min duration / 30s slide) produce a continuously updated ranking. Every 30 seconds, a new window covering the last 10 minutes is emitted. Consecutive windows heavily overlap, so rankings change smoothly rather than jumping at fixed boundaries. This is ideal for "trending now" use cases where readers want a stable, continuously refreshed list.

Tumbling windows (1min) produce non-overlapping, discrete snapshots. Each window is completely independent — there's no overlap with the previous window. This is ideal for spike detection: you compute the edit count for each discrete 1-minute bucket, compare it to a historical moving average, and flag any bucket that exceeds the threshold. The discrete boundaries make baseline computation clean and reproducible.

Q How does mapGroupsWithState work, and when would you use it over window aggregation?HARD ▼
ANSWER

mapGroupsWithState gives you full, arbitrary control over per-key state in a streaming context. For each micro-batch, Spark groups incoming events by key, then calls your user-defined function with: (1) the current key, (2) an iterator of new events for that key in this batch, and (3) the current state object stored from previous batches. Your function updates the state and emits an output row.

Use it when: you need running totals that span beyond a fixed window, custom state expiry logic, multi-event sequences (e.g., detecting a pattern across 3 consecutive edits), or when standard window aggregations don't map to your semantics (e.g., "active editors who edited in the last N minutes but haven't edited in the last 30 seconds").

Compared to window aggregation: Window aggregations are simpler, automatically managed, and more efficiently optimized by Spark. Use mapGroupsWithState only when window semantics are genuinely insufficient — the added complexity is real.

// Simplified ActiveEditors state function signature
def updateEditorState(
  userId: String,
  events: Iterator[EditEvent],
  state: GroupState[EditorState]
): EditorSummary = {
  if (state.hasTimedOut) {
    state.remove()
    return EditorSummary(userId, active = false)
  }
  val current = state.getOption.getOrElse(EditorState(0, 0L))
  val newCount = current.editCount + events.size
  state.update(EditorState(newCount, System.currentTimeMillis()))
  state.setTimeoutDuration("10 minutes") // EventTimeTimeout or ProcessingTimeTimeout
  EditorSummary(userId, active = true, editCount = newCount)
}
Q How do you reason about state size growth and cleanup?HARD ▼
ANSWER

State size = unique key cardinality × average state object size × retention window. For the active editors tracker with 1 million distinct editors/day and a 1-hour retention, even a 64-byte state object per user means ~64 MB of state — manageable in memory but needs RocksDB for production reliability.

Four control mechanisms:

Watermarks: For window aggregations, watermarks trigger state eviction for finalized windows. Without watermarks, windows accumulate indefinitely.

State timeouts: In mapGroupsWithState, call state.setTimeoutDuration() to register a timeout. On the next batch after the timeout fires, your function is called with state.hasTimedOut = true — allowing you to emit a final value and remove the key from state.

RocksDB state store: Replaces the default in-memory HashMap state store. RocksDB spills to disk, trading memory for I/O, enabling state 10–100× larger than available heap.

Checkpoint cleanup: Old checkpoint files accumulate on HDFS/S3. Configure spark.sql.streaming.minBatchesToRetain and enable log cleanup to prevent runaway storage costs.

F/U What is the difference between flatMapGroupsWithState and mapGroupsWithState?HARD ▼
FOLLOW-UP ANSWER

Both operators maintain arbitrary per-key state, but differ in output cardinality:

mapGroupsWithState must emit exactly one output row per group per batch — even if the group received no new events (when handling timeouts). This constraint makes it easy to reason about output shape but less flexible for sparse events.

flatMapGroupsWithState can emit zero, one, or many output rows per group per batch. This is the right choice when you want to emit nothing if a key has no update, or emit multiple events from a single key (e.g., pattern detection that emits an alert per matched pattern occurrence). It also supports the Append output mode, enabling downstream joins.

Prefer mapGroupsWithState for simpler logic (always one output). Use flatMapGroupsWithState when output cardinality is variable or you need Append mode compatibility.
F/U How does Spark Structured Streaming's trigger interval affect latency, throughput, and state store performance? ▼
FOLLOW-UP ANSWER

The trigger interval controls how frequently Spark kicks off a new micro-batch. The main trigger types are:

Default (as fast as possible): New batch starts immediately after the previous one completes. Minimizes latency (~seconds) but maximizes overhead from frequent checkpoint commits and state store snapshots.

ProcessingTime("30 seconds"): Batches fire on a fixed interval. If processing takes less than 30s, Spark idles until the next trigger. If processing takes more than 30s, the next batch starts immediately (no queuing). This reduces checkpoint overhead and gives predictable batch sizing.

Continuous (experimental): Epoch-based processing targeting millisecond latency, bypassing the micro-batch model. State operations are limited.

For this pipeline, a 30-second trigger balances latency (acceptable for trending analytics) with reduced state store commit frequency. The state store takes a snapshot (RocksDB checkpoint) at the end of each batch — frequent batches means frequent snapshots, which can become the bottleneck on large state.

Q What is stored in the state for ActiveEditors and EditSpikeDetector?EASY ▼
ANSWER

ActiveEditors state (per user ID):

case class EditorState(
  editCount: Long,          // running total of edits this session
  lastActivityTimestamp: Long, // epoch ms of most recent edit
  firstSeenTimestamp: Long  // epoch ms of first edit in window
)

On each batch: increment editCount, update lastActivityTimestamp, reset timeout. On timeout: emit final summary row (userId, totalEdits, sessionDuration), remove state.

EditSpikeDetector state (per article or global):

case class SpikeDetectorState(
  recentWindowCounts: List[Long], // ring buffer of last N minute-bucket counts
  movingAverage: Double,          // precomputed to avoid O(N) recalculation
  lastUpdatedBucket: Long         // which 1-min bucket was last written
)

Each micro-batch updates the current bucket count, rotates the ring buffer, recomputes the moving average, and compares the current bucket to movingAverage × spikeMultiplier. If the ratio exceeds the threshold, a spike event is emitted.

Q How would you debug a state blow-up or memory leak?HARD ▼
ANSWER

State blow-ups typically present as: increasing batch duration, executor OOM kills, or growing checkpoint size. Debugging steps:

1. Check cardinality: Add a metric that counts distinct state keys per batch. An unbounded increase indicates keys are never being evicted — check timeout logic and watermark advancement.

2. Inspect watermark progress: In the Spark UI (Streaming tab), check if the eventTime watermark is advancing. A frozen watermark (e.g., due to a partition with no new data) prevents EventTimeTimeout from ever firing.

3. Sample checkpoint files: Checkpoint state is stored as binary files in the configured checkpoint directory. Tools like spark-shell with the StateStore API can enumerate keys and values for inspection.

4. Enable state store metrics: Set spark.sql.streaming.stateStore.metrics.reportMaximumMemory=true and watch stateRows and memoryUsedBytes in the Spark UI metrics.

5. Switch to RocksDB: If state is legitimately large, move from the default in-memory store to RocksDB state store to push the problem from OOM to slower disk I/O, giving time to diagnose root cause.

A common root cause: using ProcessingTimeTimeout but the timeout duration is too long, allowing millions of keys to accumulate before any are evicted. Always profile state size during load testing before production deployment.
F/U If you needed to change the state schema (e.g., add a new field to EditorState), how would you handle the migration?HARD ▼
FOLLOW-UP ANSWER

Changing the state schema is one of the most painful operations in stateful streaming. The checkpoint stores serialized state objects — if the schema changes incompatibly, Spark will throw a deserialization error on restart.

Safe approach — additive changes with default values: Add the new field with a default value. Use a custom encoder (Kryo or Java serialization) rather than Spark's default encoder to allow versioned deserialization: check the record version field, deserialize old format records with defaults, serialize new records with the new format.

Nuclear option — checkpoint wipe: Clear the checkpoint directory and restart the pipeline from the latest Kafka offset. State is lost — all in-progress sessions expire. Only acceptable if losing a few minutes of session state is tolerable.

Blue/green migration: Run the new pipeline in parallel reading from the same Kafka topic, with a separate checkpoint directory. Let it build up fresh state over one full session window. Then cut over downstream consumers from the old Delta table to the new one.

F/U How would you implement exactly-once state updates when the output sink doesn't support idempotent writes? ▼
FOLLOW-UP ANSWER

Without an idempotent or transactional sink, the challenge is that a task can succeed in writing output but fail before committing the checkpoint — causing a replay that re-writes the same data.

Option 1 — Upsert semantics: If the sink supports upserts (e.g., a database with INSERT ... ON CONFLICT UPDATE), use the batch ID + key as the deduplication key. Replayed writes update rather than insert, making them idempotent.

Option 2 — Two-phase write: Write output to a staging table first, then atomically move it to the target table using a flag column or partition swap. On replay, overwrite the staging partition — the final move is safe to repeat.

Option 3 — Foreachbatch with manual deduplication: Use foreachBatch to write output. Inside the function, first check if the batch ID was already written (stored in a metadata table), and skip if so. This adds one extra read per batch but gives correct semantics for any sink.

Q What guarantees does Delta provide for streaming writes?MEDIUM ▼
ANSWER

Delta Lake provides two critical guarantees for streaming writes:

1. Atomic commits: Each Spark micro-batch writes data files and then atomically appends a commit entry to the Delta transaction log (_delta_log/). Readers either see the entire batch or none of it — there's no "partial batch" visible state.

2. Idempotent batch commits: Each Spark micro-batch is assigned a monotonically increasing batch ID stored in the checkpoint. When Delta writes the commit, it embeds this batch ID. On replay after a crash, Spark attempts to re-commit the same batch ID. Delta detects the duplicate and skips the write — preventing duplicate records even if the crash happened after data files were written but before the commit completed.

// Delta transaction log structure
_delta_log/
  00000000000000000000.json  ← batch 0 commit
  00000000000000000001.json  ← batch 1 commit
  00000000000000000010.checkpoint.parquet ← compacted checkpoint every 10 commits
Q How do checkpoints and Delta transaction logs work together?HARD ▼
ANSWER

They serve complementary roles in crash recovery:

Spark checkpoints track the source state: which Kafka offsets have been read, what in-memory streaming state (for stateful queries) has been accumulated, and what batch ID is next. This is Spark's "where to resume reading from."

Delta transaction log tracks the sink state: which batch IDs have been committed to the Delta table, what data files exist, and what the current table schema is. This is Delta's "what has been written."

On crash recovery: Spark reads the checkpoint to determine the last committed Kafka offset and the next batch ID to process. It re-reads from Kafka starting at that offset and writes to Delta with that batch ID. Delta consults its log, sees the batch ID is new, and commits the data. The two systems converge on a consistent state without double-writing.

Checkpoint and Delta table paths must always be co-located in terms of lifecycle. Deleting the checkpoint without rolling back the Delta table (or vice versa) puts the pipeline in an inconsistent state.
Q How do you handle schema evolution?MEDIUM ▼
ANSWER

Delta Lake provides three schema evolution modes:

mergeSchema=true: New columns in the write are added to the Delta table schema. Existing rows get null for the new column. Backward compatible — existing readers continue to work. Use for additive changes.

overwriteSchema=true: Completely replaces the Delta table's schema with the write schema. Existing data is invalidated from a schema perspective (though physically retained under time travel). Use only for full table rewrites.

Schema enforcement (default): Any write whose schema doesn't match the existing table schema fails with an AnalysisException. This is the safest default — it forces explicit acknowledgment of schema changes.

For this pipeline: add mergeSchema=true as a streaming write option. Coordinate with upstream teams using a schema registry to gate promoted schema changes. Version downstream consumers to handle both old and new schemas during the transition window.

F/U How does Delta Lake's VACUUM command work, and what's the risk of running it aggressively?MEDIUM ▼
FOLLOW-UP ANSWER

VACUUM permanently deletes data files that are no longer referenced by the current Delta table version and are older than the retention threshold (default 7 days). It reclaims storage from old versions, deleted records, and replaced files from OPTIMIZE.

Risk of aggressive vacuuming: Delta's time travel queries (VERSION AS OF, TIMESTAMP AS OF) rely on the historical data files that VACUUM deletes. After VACUUM, you can no longer query table states older than the retention window. If your SLA for data recovery or audit trails exceeds your vacuum retention, you'll silently lose that capability.

For streaming pipelines specifically: If your Spark checkpoint points to a Delta batch ID that references files deleted by VACUUM, crash recovery may fail. Always set VACUUM retention to at least 2× your maximum pipeline downtime window.

F/U What is the small file problem in Delta Lake and how do you solve it for a high-frequency streaming job? ▼
FOLLOW-UP ANSWER

Spark Structured Streaming writes one or more Parquet files per micro-batch per partition. A pipeline with a 30-second trigger and 8 output partitions generates 16+ files per minute — nearly 1,000 files per hour. Each file adds metadata overhead to the Delta transaction log and forces readers to open many small files, degrading scan performance.

Solutions:

Auto-optimize (write-time): Enable delta.autoOptimize.optimizeWrite=true on the table. Delta coalesces output within each micro-batch to produce fewer, larger files. Adds slight latency to each batch but dramatically reduces file count.

Auto-compact (post-write): Enable delta.autoOptimize.autoCompact=true. After each write, Delta checks if compaction is needed and runs OPTIMIZE asynchronously in the background.

Scheduled OPTIMIZE + ZORDER: Run OPTIMIZE table ZORDER BY (eventTime) in a nightly maintenance job to compact small files and cluster data by query patterns for faster reads.

Q What happens on a driver crash mid-batch?MEDIUM ▼
ANSWER

A Spark driver crash mid-batch leaves the pipeline in one of two states:

Scenario A — crash before Delta commit: Data files may have been written to the staging location but the transaction log commit was never made. On restart, Spark reprocesses the same Kafka offset range with the same batch ID. Delta writes the data files again (possibly overwriting the partial first attempt) and commits the transaction log entry. The result is a complete, exactly-once batch.

Scenario B — crash after Delta commit but before checkpoint commit: Delta has the batch committed; Spark's checkpoint does not reflect this. On restart, Spark attempts to re-run the same batch ID. Delta detects the duplicate batch ID in its transaction log and skips the write. The checkpoint is then updated. No duplicates, no data loss.

This two-layer commit protocol (Delta first, checkpoint second) is precisely why the combination gives exactly-once semantics even in the face of arbitrary driver crashes.

// Simplified recovery sequence
1. Spark reads checkpoint → last batch ID = 42, last offset = {p0: 1000, p1: 950}
2. Spark reads Kafka from those offsets → new events
3. Spark writes data files to Delta staging
4. Delta commits transaction log: batch 43 committed
5. Spark writes checkpoint: batch 43, new offsets
// If crash at step 4 before step 5: step 4 repeats on restart, Delta dedupes
F/U How would you validate that your pipeline is truly delivering exactly-once, not just claiming it?HARD ▼
FOLLOW-UP ANSWER

Claiming exactly-once is easy; proving it requires deliberate testing:

1. Chaos testing: Write a test harness that kills the Spark driver at random points mid-batch (e.g., using SIGKILL). Restart it and verify: row counts match the expected number of Kafka messages consumed, no duplicate event IDs exist in Delta, and no events are missing between first and last produced Kafka offset.

2. End-to-end count reconciliation: Embed a unique event counter in each SSE event (or use the Kafka offset as a unique ID). After pipeline completion, verify that every Kafka offset appears exactly once in the Delta table — no gaps, no duplicates.

3. Delta history audit: Use DESCRIBE HISTORY table to inspect all committed transactions. Cross-reference batch IDs in the Delta log with Spark checkpoint batch IDs — they should form a contiguous, non-duplicated sequence.

4. Duplicate detection query:

-- Check for duplicate event IDs in Delta
SELECT event_id, COUNT(*) as cnt
FROM delta.`/path/to/table`
GROUP BY event_id
HAVING cnt > 1
F/U When would you deliberately choose at-least-once over exactly-once?MEDIUM ▼
FOLLOW-UP ANSWER

Exactly-once semantics come with real costs: higher latency (transaction overhead), reduced throughput (sync commit points), and operational complexity (managing transactional producers, compatible sinks). In several scenarios, at-least-once is the better engineering trade-off:

1. Idempotent downstream processing: If your sink is a counter or aggregation (e.g., incrementing a page view count), duplicates cause overcounting. But if the sink is a set membership check, a duplicate is harmless. At-least-once with idempotent consumers can be simpler than full transactional exactly-once.

2. Very high throughput, low latency requirements: At-most-once (never retry) and at-least-once (retry without dedup) pipelines can achieve sub-10ms end-to-end latency; transactional exactly-once adds 10–100ms of overhead per batch.

3. Best-effort analytics: For dashboards where a ~0.01% duplicate rate is statistically invisible, the simplicity of at-least-once plus periodic reconciliation is far easier to operate than a full exactly-once stack.

Operations, Monitoring & Code
Debugging, tuning, testing, observability
8 questions
Q How do you validate data is flowing end-to-end?EASY ▼
ANSWER

A multi-layer validation approach catches issues at each stage:

Kafka: Monitor consumer lag (kafka-consumer-groups.sh --describe or via Prometheus). Zero lag on the Spark consumer group means Spark is keeping up with the producer. Growing lag indicates Spark is falling behind or has stopped.

Spark UI: Check the Streaming tab for active queries, input rows per second, batch duration, and processing rate. A healthy pipeline shows stable, bounded batch duration.

Delta: Periodically query SELECT COUNT(*), MAX(eventTime) FROM delta_table WHERE eventTime > current_timestamp() - INTERVAL 5 MINUTES. A non-zero recent count confirms data is flowing.

End-to-end heartbeat: Produce a synthetic "canary" event to the Wikimedia topic every minute with a known event ID. Assert it appears in Delta within 2× your expected batch latency. Alert if it doesn't.

Q What would cause "no data" in Delta even though Kafka has messages?MEDIUM ▼
ANSWER

In order of likelihood:

1. Filter dropping all events: The Spark query filters for type == "edit". If the Wikimedia feed changes the event type field name, all events are dropped. Check: temporarily remove the filter and count raw events from Kafka.

2. Schema parse failure: If from_json fails to parse an event (due to a schema mismatch), it returns null for all fields. A downstream filter(isNotNull(...)) then drops the row silently. Check: add a count of null-field rows before the filter as a monitoring metric.

3. Spark query not started / stuck: Check the Spark UI — is the streaming query in "ACTIVE" state? A stuck driver (e.g., awaiting a state store lock) will have no new batches despite Kafka lag growing.

4. Mismatched topic / consumer group config: Verify subscribe topic name matches exactly (case-sensitive), and the consumer group ID matches what you expect. A new consumer group with startingOffsets=latest will miss all historical messages.

5. Checkpoint corruption: If the checkpoint metadata is corrupted, Spark may fail to read the starting offset and silently start from latest, skipping all historical data. Force a checkpoint clear and restart to diagnose.

Q How do you tune spark.sql.shuffle.partitions for local vs production?EASY ▼
ANSWER

This parameter controls the number of partitions created after a shuffle (joins, aggregations). The default is 200, which is wildly oversized for local development and often undersized for large clusters.

Local development: Set to 4–8 (near your CPU core count). With 200 shuffle partitions on a 4-core laptop, each micro-batch spawns 200 tiny tasks — the scheduler overhead often exceeds the actual compute time, making batches 5–10× slower than necessary.

Production: Target 2–3 tasks per executor core at the expected shuffle data volume. Formula: total_shuffle_data_GB × 1000 / target_partition_size_MB. For a 10 GB shuffle on a 20-core cluster, target 200–400 partitions.

Adaptive Query Execution (AQE): In Spark 3+, enable spark.sql.adaptive.enabled=true. AQE dynamically coalesces shuffle partitions based on actual data skew and size, reducing the need to manually tune this parameter.

Q Why keep the Spark StructType schema next to the case class?EASY ▼
ANSWER

The StructType schema and the Scala case class are two representations of the same data contract. Keeping them in the same file (or companion object) has practical benefits:

Single source of truth: A developer changing the case class immediately sees the schema definition. Without co-location, the schema can drift: the case class gains a field, the StructType is forgotten, and from_json silently drops the new field because it's not in the schema.

Validation: In CI, add a test that derives a schema from the case class using Encoders.product[CaseClass].schema and asserts it matches the hand-written StructType. This catches drift automatically.

Documentation: The co-located definitions serve as inline documentation for both the schema structure and the type-safe Scala representation that downstream code uses.

Q How do you monitor lag and throughput?MEDIUM ▼
ANSWER

Kafka consumer lag: The most critical metric. Expose via JMX (kafka.consumer:type=consumer-fetch-manager-metrics,records-lag-max) and scrape with Prometheus. Alert when lag exceeds a threshold (e.g., 60 seconds of data at expected throughput). Use Kafka's own kafka-consumer-groups.sh for manual inspection.

Spark Streaming metrics: Spark exposes per-query metrics via the Streaming Query Listener API. Key metrics: inputRowsPerSecond, processedRowsPerSecond, batchDuration, stateOperators[*].numRowsUpdated. Expose to Prometheus via the JMX exporter or Spark's built-in Prometheus sink (spark.metrics.conf).

Delta write latency: Track the time between the event's eventTime and when it appears in the Delta table. A growing gap indicates end-to-end latency is increasing — useful for SLA monitoring.

Grafana dashboard panels: Kafka lag (per partition), Spark batch duration P50/P99, input vs processed rows/sec, state store size over time, Delta table row growth rate.

F/U If batch duration keeps increasing over time, how would you diagnose and fix it?HARD ▼
FOLLOW-UP ANSWER

Increasing batch duration is a classic symptom of one of three root causes:

1. Growing state store: More unique keys in mapGroupsWithState means more state to scan and checkpoint each batch. Diagnose with state row count metrics. Fix: tighten timeouts, use RocksDB state store, reduce key cardinality by bucketing (e.g., hourly buckets instead of per-minute).

2. Small file accumulation in Delta: As the Delta table grows, each write triggers a transaction log scan that becomes slower. Diagnose with Delta DESCRIBE HISTORY — look at the number of files in recent commits. Fix: run OPTIMIZE + VACUUM to compact files, enable auto-optimize.

3. Checkpoint I/O bottleneck: On object storage (S3/GCS), checkpoint writes can become slow as checkpoint directories accumulate thousands of files. Diagnose by timing the checkpoint write phase in Spark's task timeline. Fix: tune spark.sql.streaming.minBatchesToRetain, use a faster storage tier for checkpoints, or increase trigger interval to reduce checkpoint frequency.

Add a 30-day trailing chart of batch duration P99 to your Grafana dashboard. A slow upward trend (not visible day-to-day) often indicates state or file accumulation that will eventually cause a production incident.
F/U How would you add a new analytics job without breaking existing ones? ▼
FOLLOW-UP ANSWER

Safety comes from isolation at every layer:

Separate checkpoint path: Each streaming query must have its own unique checkpoint directory. Sharing a checkpoint directory between two queries is unsupported and will corrupt state.

Separate output path: Write to a new Delta table or a new partition scheme. Never write a new query's output to an existing table unless you've explicitly designed for multi-writer scenarios with Delta's concurrent write support.

Schema-safe transformations: The new query can reference columns from the shared parsed DataFrame, but should not rename, drop, or change the type of columns that existing queries depend on. If structural changes are needed, create a new derived DataFrame.

Resource isolation: New streaming queries consume additional executor cores and memory. On a shared cluster, validate that the new job's resource footprint doesn't starve existing jobs. Use Spark's Fair Scheduler pools to give each query a minimum resource allocation.

Feature flags: Gate the new query behind a config flag so it can be quickly disabled in production if it introduces instability.
